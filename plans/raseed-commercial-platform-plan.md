# خطة Raseed التجارية متعددة المحلات

آخر تحديث: 2026-07-13

## الرؤية

Raseed منتج Windows محلي يعمل Offline لكل محل، ويزامن فقط ما يحتاج الإنترنت: التفعيل، حالة الاشتراك، تحديثات البرنامج، والنسخ الاحتياطي المشفر. بيانات البيع اليومية وقاعدة PostgreSQL تظل على جهاز المحل، فلا يتوقف الكاشير عند انقطاع الإنترنت.

المنتج يتكون من جزأين منفصلين:

1. **Raseed Store**: التطبيق المحلي، PostgreSQL، البيع والمخزون والطباعة والمستخدمون.
2. **Raseed Control**: منصة المالك المطور لإدارة العملاء والأجهزة والتراخيص والخطط والتحديثات وحالة آخر backup، من دون استخدام حساب Super Admin ثابت داخل قواعد العملاء.

## قرارات المنتج الأساسية

- Windows 10/11 x64 فقط في الإصدار الأول.
- أول إصدار تجاري مخصص لمحلات الملابس.
- الأنشطة الأخرى تضاف كـ Business Profiles فوق Core مشترك، وليس بنسخ منفصلة من الكود.
- كل محل له قاعدة محلية مستقلة وترخيص مستقل وStore ID غير قابل للتكرار.
- لا تُشحن حسابات الديمو أو كلمات مرور ثابتة إلى أي عميل.
- لا يعتمد البيع على الإنترنت.
- لا يُحذف أو يُقفل وصول العميل إلى بياناته عند انتهاء الاشتراك؛ تتحول الوظائف التجارية حسب سياسة واضحة مع مهلة Offline.

## تجربة التثبيت المطلوبة

### ما يستلمه العميل

ملف واحد باسم قريب من `RaseedInstaller.exe`، موقّع رقميًا، أو فولدر Release يحتوي:

- Bootstrapper موقّع.
- manifest بإصدارات كل المتطلبات وروابطها وSHA-256.
- Full offline bundle احتياطي عند تعذر الإنترنت.
- تعريفات الطابعات المعتمدة فقط، وليس أي تعريف عشوائي من الإنترنت.
- دليل استلام مختصر وملف release notes.

### ما يفعله الـ Bootstrapper

1. يتأكد من Windows x64، صلاحيات Administrator، المساحة، والإنترنت.
2. يكتشف PostgreSQL الموجود ويحدد إن كان إصدارًا مدعومًا.
3. إن لم يوجد، ينزل إصدار PostgreSQL مثبت ومحدد مسبقًا من مصدر موثوق، ويتحقق من التوقيع والـ checksum، ثم يشغله silent.
4. ينشئ Windows Service تلقائية التشغيل، database، ومستخدمًا محدود الصلاحيات بكلمة مرور عشوائية.
5. يحفظ connection والأسرار باستخدام Windows DPAPI/safeStorage.
6. يثبت Raseed per-machine ويشغل migrations وhealth checks.
7. يختبر الكتابة والقراءة في قاعدة البيانات.
8. يفتح Device Setup: اسم المحل، النشاط، الطابعة، مقاس الورق، السكانر، ومجلد backup.
9. يطلب Claim Code أو License Code صادرًا من Raseed Control.
10. ينشئ أول Owner حقيقي ثم يحذف setup token نهائيًا.
11. يطبع Test Receipt ويجري Test Scan قبل إعلان نجاح التثبيت.
12. ينتج Installation Report لا يحتوي كلمات مرور.

لا يتم تنزيل "آخر إصدار" مباشرة وقت التثبيت؛ يتم تنزيل إصدار pinned من manifest موقّع تملكه Raseed لمنع كسر العملاء أو هجمات supply chain.

## الحسابات والصلاحيات

### حسابات الديمو

- حسابات الكاشير والـ Owner الحالية تبقى في Demo build فقط.
- Production seed يحتوي permissions والـ system roles فقط، ولا يحتوي متاجر أو مستخدمين أو فواتير تجريبية.
- Demo database لا تتحول إلى قاعدة عميل.

### أول عميل جديد

1. تنشئ العميل والمتجر والخطة من Raseed Control.
2. المنصة تصدر Claim Code قصير العمر ومستخدمًا مرة واحدة.
3. أثناء التثبيت تدخل بيانات الـ Owner الحقيقي: الاسم، الهاتف، وكلمة مرور يختارها العميل.
4. الـ Owner ينشئ حسابات الكاشير ويحدد الفرع والصلاحيات.
5. لو أنت ستجهز الحسابات، تستخدم Technician Session مؤقتة ومسجلة تنتهي بعد التسليم، وليس Super Admin دائمًا.
6. العميل يغير كلمة مرور الـ Owner عند الاستلام ويوقع على نموذج الاستلام.

### Super Admin

- Super Admin يخص Raseed Control السحابي فقط.
- لا يوجد global backdoor password داخل أجهزة المحلات.
- الدعم البعيد يتطلب موافقة العميل، كود جلسة مؤقت، وسجل audit واضح.

## الترخيص ونماذج البيع

### شهري وسنوي

- License lease موقعة بمفتاح asymmetric؛ المفتاح الخاص عندك فقط، والعام داخل التطبيق.
- الجهاز يجدد lease عند توفر الإنترنت.
- مهلة Offline مقترحة 14 يومًا بعد موعد التجديد لتجنب تعطيل محل بسبب انقطاع مؤقت.
- منع التلاعب بالساعة وتسجيل آخر وقت موثوق، مع سياسة لا تفقد بيانات العميل.

### بيع كامل

- Perpetual license للمحل أو لعدد أجهزة محدد.
- استمرار الاستخدام بلا انتهاء، مع فصل "حق الاستخدام" عن "سنة التحديثات والدعم والـ Cloud Backup".
- نقل الترخيص لجهاز بديل يتم من المنصة بعد إلغاء الجهاز القديم أو موافقة دعم موثقة.

### الخطط المقترحة

- `Starter`: جهاز واحد، فرع واحد، POS ومخزون وbackup محدود.
- `Business`: عدة مستخدمين وتقارير وموردون ومشتريات وCloud Backup كامل.
- `Multi-Branch`: عدة أجهزة/فروع ومزامنة مركزية، وهذه مرحلة لاحقة لأنها ليست نفس معمارية قاعدة محلية واحدة.
- `Perpetual`: استخدام دائم مع عقد منفصل للتحديث والدعم والتخزين السحابي.

## Offline وCloud Backup

### قواعد التشغيل

- كل عمليات POS والمخزون والشيفت تعمل محليًا بلا إنترنت.
- Agent داخل التطبيق يراقب الاتصال ولا يعطل الواجهة.
- عند عودة الإنترنت يرفع آخر backup ناجح وmetadata فقط: Store ID، إصدار schema، وقت النسخة، الحجم، والـ checksum.
- retries مع exponential backoff ورفع قابل للاستكمال.
- لا تُرفع قاعدة live مباشرة؛ ينشأ dump متسق أولًا ثم يضغط ويشفر محليًا.

### التشفير والاستعادة

- مفتاح تشفير مختلف لكل متجر.
- backup مشفر قبل مغادرة الجهاز باستخدام AES-256-GCM.
- TLS أثناء الرفع، وتخزين Object Storage مع versioning وretention.
- Recovery Key يطبع/يسلم للعميل، مع نسخة recovery مشفرة لديك وفق العقد حتى لا يصبح تلف الجهاز نهاية البيانات.
- retention مقترح: 7 يومية + 4 أسبوعية + 12 شهرية، قابل للتغيير حسب الخطة.
- حذف العميل أو انتهاء الخطة له سياسة احتفاظ وإخطار واضحة، وليس حذفًا فوريًا.
- Restore drill إلزامي؛ نجاح upload وحده لا يثبت صلاحية النسخة.

### لوحة Raseed Control

- آخر جهاز متصل وآخر backup ناجح وحجمه وإصدار التطبيق.
- تنبيه لو لم تصل نسخة خلال 24/48 ساعة.
- إصدار/تجديد/إلغاء/نقل التراخيص.
- إدارة خطط العملاء والمدفوعات يدويًا في البداية ثم بوابة دفع لاحقًا.
- نشر تحديثات بقنوات `pilot`, `stable` مع rollback.
- لا تعرض مبيعات العميل التفصيلية افتراضيًا؛ أقل قدر بيانات مطلوب للدعم.

## الأجهزة والطباعة

### Barcode Scanner

- الاعتماد أولًا على USB scanners التي تعمل Keyboard HID ولا تحتاج driver خاصًا.
- شاشة اختبار تعرض آخر barcode وسرعة القراءة وتمنع Enter المكرر.
- دعم prefixes/suffixes من الإعدادات لاحقًا.

### Receipt Printer

- قائمة موديلات معتمدة 58mm و80mm يتم اختبارها فعليًا.
- Device Profile لكل موديل: driver، اسم الطابعة، المقاس، margins، cut، cash drawer، وعدد النسخ.
- المثبت ينزل driver فقط إذا كان من مصدر الشركة وتم تثبيت checksum/version.
- fallback إلى Windows Print Dialog للطابعات غير المعتمدة، من دون وعد بالطباعة الصامتة.

### Barcode Label Printer

- مرحلة منفصلة عن receipt printer.
- قوالب ملابس بالمقاسات الفعلية، اختبار scan بعد الطباعة، وإعداد DPI لكل موديل.

## Business Profiles

### Clothing — الإصدار الأول

- Product variants: لون، مقاس، موسم، خامة، ماركة.
- barcode لكل variant، طباعة labels، مصفوفة stock لون × مقاس.
- استبدال ومرتجع، خصومات، حجز فاتورة، عميل اختياري.
- موردون وأوامر شراء واستلام variants.
- تقارير الأكثر بيعًا، الراكد، الربح، المخزون حسب المقاس واللون.

### Supermarket — بعد استقرار الملابس

- weighted/price barcodes، وحدة ووزن، ميزان، صلاحية وbatch.
- سرعة POS أعلى، عروض متعددة الشروط، ضرائب، موردون واستلام سريع.
- دعم أجهزة الميزان يحتاج تكاملًا واختبارات hardware مستقلة.

### Bookstore/Stationery

- ISBN، مؤلف، ناشر، طبعة للكتب.
- وحدات/ألوان/ماركات للأدوات المكتبية.
- بحث ISBN وإمكانية استيراد catalog من مصدر قانوني لاحقًا.

### قاعدة التصميم

كل profile يحدد الحقول والـ navigation والتقارير والـ defaults، بينما البيع والمستخدمون والمدفوعات والمخزون الأساسي مشتركون. لا نظهر خصائص السوبرماركت لمحل الملابس.

## Playbook أول بيع لمحل الملابس

1. جرد الأجهزة وموديلات الطابعة والسكانر وWindows قبل الموعد.
2. إنشاء العميل والخطة والترخيص من Raseed Control.
3. Backup لصورة Windows/restore point ثم تشغيل المثبت الموقّع.
4. إنشاء Owner الحقيقي، ثم حسابات الكاشير بأسماء حقيقية وصلاحيات محدودة.
5. ضبط الفروع والإيصال والطابعة والسكانر والباركود.
6. استيراد المنتجات بعد preview وموافقة العميل.
7. Test shift و10 فواتير تشمل خصمًا ودفعًا مختلطًا ومرتجعًا.
8. جرد عينة ومطابقة الفواتير والمدفوعات والمخزون.
9. إنشاء backup محلي وسحابي واستعادته على قاعدة اختبار.
10. تسليم Recovery Key وكلمات المرور والعقد ودليل الدعم، وتوقيع محضر الاستلام.
11. متابعة يومية أول أسبوع، ثم أسبوعية أول شهر.

## قرارات مطلوبة من مالك Raseed

هذه لا تمنع بدء Phase 0 و1، لكنها مطلوبة قبل بناء Raseed Control:

1. البيع الكامل: هل الترخيص لجهاز واحد أم لمحل مع عدد أجهزة؟ التوصية: للمحل مع device limit مكتوب.
2. مهلة Offline للاشتراك: التوصية 14 يومًا.
3. Recovery Key: هل تريد قدرة دعم لاستعادة backup بموافقة العميل؟ التوصية: نعم بمفتاح recovery مشفر ومراجَع.
4. بلد الإطلاق ومتطلبات الفاتورة والضرائب؛ الخطة تفترض مصر لكن يلزم تحديد هل أول عميل يحتاج فاتورة ضريبية/إيصال فقط.
5. موديل الطابعة والسكانر لأول محل؛ يجب شراؤهما/معرفتهما قبل اعتماد Driver Profile.
6. اسم الشركة والدومين والبريد الرسمي وشهادة Code Signing ومكان استضافة Raseed Control.

## خارج الوعد في أول إصدار

- تثبيت تلقائي مضمون لكل طابعة أو ميزان في السوق.
- عدة فروع تعمل على نفس البيانات عبر الإنترنت.
- إغلاق البرنامج نهائيًا عند انقطاع الإنترنت.
- نسخ كل بيانات العميل التحليلية إلى السيرفر من دون عقد وموافقة.
- دعم كل أنواع المحلات قبل نجاح Clothing Pilot.

## مصادر القرار التقني

- Electron Builder يدعم NSIS وNSIS Web وتخصيص installer وإضافة prerequisite installers مع checksum للـ web payload: https://www.electron.build/nsis/
- Windows DPAPI مخصص لحماية الأسرار بمستخدم/جهاز Windows بدل شحن مفتاح ثابت: https://learn.microsoft.com/en-us/windows/win32/seccng/cng-dpapi
